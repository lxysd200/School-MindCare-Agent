# 关键决策记录（Decisions）

目标：把影响架构、接口与长期维护成本的关键选择记录下来，方便复盘与对外展示工程思维。

## 记录格式（ADR 简化版）

- 背景：为什么要做这个选择
- 方案：有哪些可选项
- 决策：最终选了什么
- 影响：带来的收益与代价
- 后续：需要补的工作与风险

---

## D001：使用 LangGraph 编排多阶段 Agent 流程

- 背景：需要把“记忆→意图→知识→风险→回复”拆成可观测、可测试的步骤。
- 方案：手写状态机 / LangChain LCEL / LangGraph。
- 决策：选择 LangGraph，使用 StateGraph + 条件路由。
- 影响：流程可视为图结构，后续加 RAG/MCP 节点更自然；代价是需要维护状态结构与节点边界。
- 后续：补充统一的 step 追踪输出与可视化，建立回归用例。

---

## D002：LLM 后端采用统一 OpenAI SDK 调用接口

- 背景：需要同时支持 OpenAI / DeepSeek / Ollama，降低切换成本。
- 方案：每家 SDK 单独封装 / 统一走 OpenAI 兼容接口。
- 决策：统一走 OpenAI SDK 的 chat.completions 调用方式，通过 base_url 与 api_key 切换后端。
- 影响：适配层更薄，配置更简单；代价是受限于 OpenAI 兼容程度，需要处理各后端差异。
- 后续：补充错误分类、重试与超时策略；对模型输出做更严格的结构化校验。

---

## D003：记忆压缩采用“近期原文 + 中期摘要”混合注入

- 背景：上下文窗口有限，需要在低成本下兼顾长期信息与最近对话一致性。
- 方案：全量拼接 / 向量检索 / 规则抽取 / LLM 摘要。
- 决策：先实现“近期原文 + 中期摘要”混合策略，并提供 token budget 控制。
- 影响：工程实现简单、可解释；代价是对“语义相关性”支持有限，后续需要 RAG/向量检索增强。
- 后续：把记忆数据源从本地文件抽象出来，支持持久化与检索。
